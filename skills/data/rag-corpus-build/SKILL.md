# SKILL: rag-corpus-build — 로컬 RAG 인덱스 구축 (점유·비용 민감)

## Trigger
텍스트 코퍼스로 검색용 벡터 인덱스를 만들 때. "RAG 구축", "임베딩 색인", "벡터 DB". 특히 로컬 점유·비용을 낮춰야 할 때.

## Procedure
1. **필드 정제**: 원자료에서 실제 검색에 쓸 자연어만 추출. 단 **케이스를 구별하는 핵심 메타(연료·규모·주제 등)는 버리지 말고 청크 글 머리에 헤더로 넣는다**(검색이 그 축으로 맞아 들어감). 순수 수치·구조 필드만 필터용 메타로 분리.
2. **데이터 중복도 먼저 측정 (신설·필수)**: 색인·평가 전에 코퍼스가 정형/합성인지 본다 — 고유 첫줄 수·고유 줄 비율. 중복이 심하면(예: 한 질문이 수백 케이스에 반복) 케이스가 서로 구별 안 됨 → 청킹·평가 전략이 달라진다.
3. **청킹 선택**: 케이스가 잘 구별되면 구조 단위(줄/문장). **정형·중복 심하면 케이스당 글 하나(헤더+대표 문장, "casehdr")** — 조각 수가 케이스 수로 급감(압축이 불필요해짐).
4. **평가셋 — 무효 함정 주의**: known-item 자동셋(질문→그 케이스)은 **질문이 여러 케이스에 중복되면 무효**(정답이 수백 개라 점수가 0처럼 붕괴 — 품질 아닌 시험지 문제). 그래서 ②에서 중복도 먼저 확인. 신뢰 지표 = 상황형 질의로 주제·메타 일치율 + 손으로 쓴 패러프레이즈셋. recall이 known-item 상한인데도 0에 가깝고 압축·nprobe를 바꿔도 안 변하면 → 인덱스 아닌 데이터/평가를 의심.
5. **인덱스 선택**: 조각이 적으면(수십만) **flat(압축없는 정확검색)** = 최고 품질·1GB급. 조각이 많을 때만 IVF-PQ(손실 압축). 단 PQ 강도를 정하기 전 평가셋이 유효한지부터 확인(무효 평가로 PQ 정하면 헛수고). 점유 = 청크수 × dim × 4B.
6. **전량 색인**: 확정 설정으로 오픈 임베딩(예 BGE-m3) → 인덱스 → 버킷 보관 + 질의시 온디맨드 로드. raw·정제본은 색인 후 로컬 삭제(버킷이 원본). 무거운 임베딩은 [cloud-gpu-embed] 참고(발열 회피).

## Pitfalls
- **정형/합성 데이터에서 known-item 평가셋은 "쉬운 상한"이 아니라 "무효"가 될 수 있다.** 질문 중복 → R@1이 1/중복수로 붕괴. 색인 전 중복도부터 측정. (PQ 압축 탓으로 오진해 비교 스모크 헛쓴 사례 있음.)
- **blind 512 윈도우 = 청크 폭증.** 구조적이면 구조 단위로, 중복 심하면 케이스당 1조각(casehdr).
- **flat 인덱스가 raw보다 클 수 있음**(N×dim×4B). 단 조각이 적으면 flat이 최선(압축 손실 0).
- **맥에서 torch(임베딩)+faiss(검색)를 한 프로세스에 함께 두면 libomp 충돌 세그폴트.** embed/search를 별도 하위프로세스로 분리(embed→.npy→search). + bge-m3 + torch MPS도 세그폴트 → `device="cpu"` + 스크립트 상단 `os.environ.setdefault("KMP_DUPLICATE_LIB_OK","TRUE")`(torch import 전).
- 임베딩 비용은 1회 한 자릿수~수십 달러로 작음 → 모델은 돈 아닌 품질·프라이버시로 고른다.

# 에이전트 설계 전에 알아야 할 것 — 단계별 정리

> 대상: 취업지원 Agent 같은 **단일 에이전트(LangGraph 고정 그래프)**를 직접 설계하려는 사람.
> 순서대로 따라가면 "코드 짜기 전에 무엇을 정해야 하는지"가 빠짐없이 잡힌다. 각 단계는 취업지원 Agent에 바로 대입해 설명한다.

---

## 요약

설계는 코드가 아니라 **개념 → 문제 → 데이터 → 노드 → 프롬프트 → 흐름 → 평가 → 비용 → 검증 → 구현** 순으로 정한다. 핵심은 셋이다. (1) "에이전트 = 상태 하나가 그래프를 도는 단일 루프"라는 그림을 먼저 머리에 넣는다. (2) 코드보다 **상태(데이터가 어떻게 흐르나)와 노드 경계(누가 무엇을 책임지나)**를 먼저 종이에 그린다. (3) 루프엔 반드시 **종료조건과 비용 가드**를 박는다. 아래 0~9단계가 이 순서다.

---

## 0단계 — 개념 토대 (용어부터 못 박기)

코드 전에 이 단어들의 그림이 머리에 있어야 한다.

**LLM 호출의 본질.** 호출 한 번은 "프롬프트(입력 텍스트) → 결과 텍스트"가 전부다. **무상태**다 — 이전 호출을 기억 못 한다. 그래서 맥락은 매번 프롬프트에 다시 넣어줘야 하고, **토큰만큼 과금**된다. 에이전트의 "기억"은 LLM이 아니라 우리가 들고 다니는 상태 객체가 한다.

**에이전트 vs 워크플로.** 둘 다 LangGraph로 만들지만 다르다. **자율 에이전트**는 매 단계 LLM이 "다음에 뭘 할까"를 스스로 고른다(유연하지만 비싸고 예측 어려움). **워크플로(고정 그래프)**는 순서를 사람이 미리 박는다(싸고 예측 가능). 우리 취업 Agent는 **고정 그래프**다.

**4가지 부품.** ① **상태(state)** = 노드들이 함께 쓰는 공유 메모리. ② **노드(node)** = 일 하나를 하는 함수(보통 LLM 호출 1번). ③ **엣지(edge)** = 노드에서 노드로 가는 화살표. ④ **라우터(조건부 엣지)** = "다음에 어디로 갈지"를 정하는 분기. 우리 라우터는 LLM 없이 규칙으로 도니 공짜다.

**루프.** 평가→재작성→평가처럼 같은 노드로 되돌아오는 길. 되돌아오는 길이 있으면 반드시 **언제 멈출지(종료조건)**가 짝으로 있어야 한다. 없으면 무한·비용 폭주.

---

## 1단계 — 문제 정의 (코드 한 줄 전에)

**입력과 출력을 한 문장으로.** "공고 1건 + 문항 1개 + 경험 3~4개 → STAR 자소서 1건." 이게 안 나오면 아직 설계할 게 아니다.

**성공을 측정 가능하게.** "좋은 자소서"는 측정 불가. "재작성 3회 안에 평균 4.0/5, 또는 점수가 오르다 정체 시 자동 종료"처럼 **숫자로** 바꾼다.

**범위를 한 줄기로 자른다(수직 슬라이스).** 4개 기능을 한 번에 X. 끝에서 끝까지 도는 가장 가는 줄기 하나만 먼저.

**가설 1개로 좁힌다.** "스스로 평가해 다시 쓰면 반복할수록 좋아진다" — 이 한 줄만 검증. 나머지는 이게 산 다음.

---

## 2단계 — 상태 설계 (데이터가 어떻게 흐르나)

코드보다 **이걸 먼저 종이에 그린다.**

**끝까지 흐름을 그려본다.** 입력에서 최종 출력까지 어떤 데이터가 생기고 쓰이는지 전부 적는다. 빠진 칸이 있으면 노드 사이에서 막힌다.

**각 칸의 키·타입을 못 박는다(깨끗한 형식).** `jd_analysis = {핵심역량:[문자열], 우대사항:[문자열], 인재상:[문자열]}`처럼. 줄글로 넘기면 다음 노드가 재해석하느라 호출이 또 든다.

**생산자/소비자 매핑.** 칸마다 "누가 채우고(생산) 누가 읽나(소비)"를 적는다. 예: `jd_analysis`는 분석 노드가 채우고 생성·평가 노드가 읽는다. 이게 노드 경계를 정하는 근거가 된다.

---

## 3단계 — 노드 쪼개기 (경계 정하기)

**한 노드 = 한 책임.** 경험분석, 공고분석, 생성, 평가는 각각 다른 일이니 다른 노드. 한 노드가 두 가지를 하면 프롬프트가 길어지고 디버깅이 어렵다. 반대로 너무 잘게 쪼개면 호출만 는다 — "한 가지 일"이 기준.

**노드마다 3종 세트를 적는다.** ① 입력 칸(상태의 무엇을 읽나) ② 출력 칸(상태의 무엇을 채우나) ③ 통과 기준(이 노드가 끝났다고 볼 조건). 이 3종이 곧 "노드 역할 박제"다.

**LLM 노드 vs 규칙 노드를 가른다.** 판단이 필요하면 LLM(과금), 단순 분기·계산이면 규칙(공짜). 라우터·종료조건 판정은 규칙으로 — 여기에 LLM 쓰면 돈만 샌다.

---

## 4단계 — 프롬프트 설계 (노드 내용을 도메인에 특화)

여기가 품질이 갈리는 곳. 노드 골조는 범용, **내용은 취업코칭에 깊게 특화**.

**역할·지시·형식.** "너는 채용 심사위원이다(역할), 3축으로 감점부터 찾아라(지시), `{점수,근거,개선지시}` JSON으로만 답하라(형식)." 출력 형식을 강제해야 2단계의 깨끗한 형식이 지켜진다.

**도메인 지식 주입.** STAR 4요소, 한국 공고의 우대/인재상 표현, 자소서 채점 기준 같은 걸 프롬프트에 박는다. 단 **특정 회사·공고는 박지 말 것** — 그건 입력으로 받아 분석. (도메인엔 특화, 특정 채용건엔 비특화)

**호출 파라미터.** 생성은 온도 조금 높게(다양성), 평가는 낮게(일관성). 같은 모델이라도 역할을 이렇게 분리한다.

**예시(few-shot)는 필요할 때만.** 형식이 안 잡히면 좋은 예 1~2개를 넣는다. 토큰이 느니 무조건 넣진 않는다.

---

## 5단계 — 흐름·라우팅 설계

**노드 순서(엣지)를 그린다.** 분석 → 매칭 → 생성 → 평가 → (라우터) → 재작성 → 평가…

**분기 조건을 규칙으로 적는다.** 라우터가 보는 건 점수뿐: 통과면 종료, 아니면 재작성.

**종료조건 3개를 반드시 박는다.** ① 목표 도달(평균≥4.0 & 최저≥3.5) ② 최대 횟수(3회) ③ 향상 정체(직전 대비 +0.3 미만). 루프엔 가드가 짝이다 — 무한·비용 폭주 방지.

---

## 6단계 — 평가·검증 설계 (자기개선의 심장)

**무엇을 점수화할지 못 박는다.** 논리성/직무적합성/설득력, 각 1~5점, 축마다 감점 근거 + 행동형 개선지시 한 줄. 기준이 흔들리면 루프 전체가 흔들린다.

**자기편향을 줄인다.** LLM은 자기 글을 후하게 본다. "감점부터 찾아라" + 온도↓ + 역할 분리로 줄이되, 없앨 수 없다는 걸 인정하고 사람 대조 1건을 병행.

**향상을 추적한다.** 회차별 점수를 history에 쌓아 곡선을 본다. 이게 "정말 좋아지는가"를 보여주는 증거이자 정체 종료의 근거.

---

## 7단계 — 비용·모델 설계

**호출 수를 센다.** 자소서 1건 = 분석1+매칭1+생성1+(평가+재작성)×최대3 ≈ 최대 8호출. 라우터는 공짜.

**모델 계층화.** PoC는 싼 모델 하나로 전부. 품질 미달이면 **생성 노드만** 상위 모델로 올린다. 처음부터 비싼 모델 X.

**과금 구조가 바뀌는 지점을 미리 표시.** 검색 API(호출당 과금)·상시 서버(시간당 호스팅)·유료 트레이스 — 이 셋에 들어가는 순간 유휴 0이 깨진다. PoC에선 셋 다 안 쓴다.

---

## 8단계 — 검증 계획 (만들기 전에 어떻게 확인할지 먼저 정한다)

**결정적 부분은 공짜로.** 상태 배선·라우팅·종료조건은 가짜 점수를 주입한 **오프라인 스텁**으로 LLM 없이 돌려 확인한다. 깨끗한 형식 덕에 가능.

**LLM 노드는 실제로 최소 1회.** 끝에서 끝까지 e2e 1회 돌려 점수 곡선을 본다.

**골든 기준 1건.** 최종 자소서 1건을 사람 눈으로 합격선 판단. 자동 점수만 믿지 않는다.

---

## 9단계 — 구현·졸업 경로

**노트북에서 시작.** 셀 단위로 노드를 하나씩 만들어 돌려본다.

**안정되면 .py 모듈로 졸업.** `state.py / prompts/ / nodes.py / graph.py / run.py`. 전부 **한 레포**(노드별 레포 X).

**노드 역할은 파일로 박제.** 노드 하나당 `프롬프트 + 입출력 형식 + 통과 기준`을 한 묶음으로 고정. 한 노드만 고칠 땐 그 파일만.

---

## 흔한 함정 (미리 알고 피하기)

- 루프에 종료조건을 안 박아 무한·비용 폭주.
- 노드 간을 줄글로 넘겨 다음 노드가 재해석(호출 낭비·오류).
- 특정 회사/공고를 노드에 하드코딩 → 다음 공고에서 깨짐.
- 측정 불가한 성공 기준("좋은 자소서") → 검증 불가.
- 처음부터 검색·서버·비싼 모델을 넣어 가설 검증 전에 비용 폭발.
- 자동 점수만 믿고 사람 대조 생략 → 자기편향 못 잡음.

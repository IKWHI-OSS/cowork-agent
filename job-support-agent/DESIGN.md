# AI 취업지원 Agent — PoC 설계 (싸게 구현·빠르게 검증)

> 목표: 채용공고와 지원자 경험을 받아 **자소서를 만들고, 스스로 평가해 다시 고치는** 단일 에이전트.
> 이 문서는 1차 PoC의 **로직 설계**다. 범위는 "수직 슬라이스 1줄기"(공고분석 → 자소서 생성 → 피드백 루프)로 잡고, 면접 지원과 세션 간 학습은 2차로 분리한다.

---

## 1. 요약 (먼저 읽기)

PoC에서 검증하려는 **핵심 가설은 단 하나**다: "에이전트가 자기 자소서를 평가하고 다시 쓰면, 반복할수록 글이 좋아진다." 이 한 줄이 사면 나머지(면접·맞춤화 확장)는 같은 골격에 노드만 더 붙이면 된다. 그래서 1차에는 가설과 무관한 비용(임베딩/검색 API/벡터DB/서버 배포)을 전부 뺀다.

구현은 단일 에이전트 + 단일 LLM이다. "분석가 → 작가 → 심사위원"은 별도 에이전트가 아니라 **같은 LLM에 역할만 바꿔 거는 프롬프트**다. 공고와 경험은 검색하지 않고 **프롬프트에 그대로 넣는다**(붙여넣기 수준). 자기개선은 생성→평가→재작성을 도는 루프인데, **종료조건 3개**(목표점수 도달 / 향상 정체 / 최대 횟수)로 비용 폭주를 막는다.

비용은 LLM 호출당 과금만 발생한다. 자소서 1건 완성에 최대 약 8회 호출(분석 1 + 생성 1 + 평가·재작성 최대 3쌍)이면 충분하다. 검증은 노트북에서 셀 단위로, 결정적 부분(상태 배선·라우팅·종료조건)은 샌드박스 오프라인 스텁으로 먼저 돌리고 실제 LLM 호출만 사용자 맥에서 한다.

---

## 2. 1차 범위 (무엇을 넣고 무엇을 뺐나)

**넣는 것 (수직 슬라이스):**
공고 텍스트 1건과 자소서 문항 1개, 사용자가 입력한 경험 3~4개를 받아서 → 공고를 분석하고 → STAR 구조로 자소서 초안을 쓰고 → 스스로 평가해 → 기준에 못 미치면 다시 쓰는, **끝에서 끝까지 도는 한 줄기**.

**2차로 미루는 것 (이유는 비용·검증 우선순위):**

- 면접 지원(예상질문·꼬리질문·모의면접): 핵심 가설이 아님. 골격이 검증되면 노드 추가로 붙는다.
- 세션 간 학습("쓸수록 성장"): 고득점 글·반복 지적을 누적해 다음 자소서에 재사용하는 부분. 가치는 있으나 1차 가설 검증엔 불필요하고 저장·관리 복잡도만 올린다.
- 검색/RAG: 공고·경험은 분량이 적어 프롬프트에 직접 넣으면 된다. 임베딩·벡터DB는 가장 비싼 단계라 가설 검증 전엔 보류한다.

---

## 3. 데이터 흐름과 상태

에이전트는 하나의 **상태 객체**를 들고 노드 사이를 이동한다(LangGraph State). 상태에 담기는 값:

```
state = {
  job_posting:      "<공고 원문 텍스트>",
  question:         "<자소서 문항, 예: 지원동기와 입사 후 포부>",
  experience_bank:  [ {situation, task, action, result}, ... ],   # 사용자 입력

  jd_analysis:      { 핵심역량:[...], 우대사항:[...], 인재상:[...] },  # 분석 노드가 채움
  matched:          [ {핵심역량: "...", 근거경험: <experience_bank의 항목> }, ... ],

  draft:            "<현재 자소서 텍스트>",
  evaluation:       { 논리성:{score,근거,개선지시},
                      직무적합성:{...}, 설득력:{...} },
  history:          [ {draft, 평균점수}, ... ],   # 정체 판단용
  iteration:        0
}
```

흐름(화살표 = 노드 이동):

```
 [입력]
   │  공고·문항·경험
   ▼
 (1) analyze_jd ──► jd_analysis 채움 (핵심역량/우대/인재상 추출)
   ▼
 (2) match_experience ──► 어느 경험이 어느 핵심역량을 증명하는지 매핑
   ▼
 (3) generate_draft ──► STAR 구조로 자소서 초안 작성
   ▼
 (4) evaluate ──► 3축 채점(1~5) + 개선지시   ◄────────┐
   ▼                                                  │
 (5) route ── 통과? ──► [완성·종료]                    │
   │  미달                                            │
   ▼                                                  │
 (6) rewrite ──► 개선지시 반영해 다시 작성 ────────────┘
```

(4)→(5)→(6)→(4)가 **자기개선 루프**다.

---

## 4. 노드별 로직 (단일 LLM, 역할만 교체)

**(1) analyze_jd — 공고 분석가.** 공고 원문을 받아 핵심역량/우대사항/인재상을 각각 항목 리스트로 뽑는다. 출력은 JSON으로 고정해 다음 노드가 바로 쓰게 한다. 인재상은 회사 소개·가치 문구에서 추출하되, 없으면 빈 리스트로 두고 무리해서 지어내지 않는다.

**(2) match_experience — 매칭.** 핵심역량 각각에 대해 experience_bank에서 가장 잘 증명하는 경험을 고른다. 경험 수가 적으니 검색 없이 LLM이 통째로 보고 순위만 매긴다. 한 핵심역량에 마땅한 경험이 없으면 "근거 부족"으로 표시해 생성 단계가 과장하지 않게 한다.

**(3) generate_draft — 작가.** 매칭 결과 + 문항을 받아 자소서를 쓴다. 각 경험은 **STAR 4요소**(상황·과제·행동·결과)를 한 문단 안에 흐르게 녹인다. 행동(Action)은 지원자가 **주도적으로 한 일** 중심, 결과(Result)는 가능하면 수치. 분량은 문항 글자수 제한을 인자로 받아 지킨다.

**(4) evaluate — 심사위원(LLM-as-judge).** 초안을 3축으로 1~5점 채점하고, 축마다 **감점 근거**와 **구체적 개선지시 한 줄**을 낸다. 채점 기준은 5절. 자기 글을 후하게 보는 편향을 줄이려고 판사 프롬프트는 "감점 사유부터 찾아라"로 엄격하게 걸고, 생성과 **온도를 낮춰 역할을 분리**한다.

**(5) route — 라우터(LLM 아님, 규칙).** 종료조건(6절)을 코드로 판정한다. 통과/최대횟수/정체 중 하나면 종료, 아니면 rewrite로 보낸다. **여기는 LLM을 안 써서 공짜**다.

**(6) rewrite — 작가(개정).** 직전 초안 + 평가의 개선지시만 받아 다시 쓴다. 전체를 새로 쓰지 말고 **지적된 부분 위주로 고치게** 지시해 토큰·비용을 아끼고 향상을 추적 가능하게 한다. 새 초안은 다시 (4)로 들어간다.

---

## 5. 평가 기준 (자기개선의 핵심)

세 축, 각 1~5점. 채점이 흔들리면 루프 전체가 흔들리므로 기준을 못 박는다.

**논리성.** STAR 4요소가 다 있는가, 상황→행동→결과의 인과가 이어지는가, 앞뒤 모순이 없는가. (구조 결손 = 큰 감점)

**직무 적합성.** jd_analysis의 핵심역량·우대사항을 **실제 경험으로 증명**했는가, 아니면 단어만 베껴 왔는가. 공고에 없는 자랑은 가점 없음, 핵심역량을 비워 두면 감점.

**설득력.** 구체적 수치·성과가 있는가, 남과 구별되는 점이 있는가, 군더더기·추상적 미사여구가 없는가.

각 축에서 LLM은 `{점수, 감점근거, 개선지시}`를 낸다. 개선지시는 rewrite가 바로 실행할 수 있게 **행동형 한 줄**("3번째 문단 결과를 수치로 바꿔라")로 강제한다.

---

## 6. 종료조건 (비용 폭주 방지)

루프는 다음 **셋 중 하나**면 멈춘다. 무한 재작성과 돈 낭비를 막는 장치다.

1. **목표 도달** — 세 축 평균 ≥ 4.0 **그리고** 최저 축 ≥ 3.5. (한 축만 좋아 평균이 뜨는 걸 막음)
2. **최대 횟수** — 재작성 3회까지. 그 이상은 보통 향상이 미미하다.
3. **향상 정체** — 직전 대비 평균 점수 향상이 0.3 미만이면 "더 돌려도 안 오른다"로 보고 종료. (history로 판단)

정체 종료는 "근거 없이 같은 시도를 반복하지 않는다"는 작업 원칙과 같은 취지다. 멈춘 뒤에는 마지막 초안과 점수 곡선을 함께 반환해, 왜 거기서 멈췄는지 보이게 한다.

---

## 7. 왜 싸고 빠른가

**비용.** 과금은 LLM 호출당만 발생한다. 자소서 1건 = 분석 1 + 매칭 1 + 생성 1 + (평가+재작성) 최대 3쌍 ≈ **최대 8호출**. 라우터는 규칙이라 공짜. 임베딩·검색 API·벡터DB·상시 서버가 없으므로 **유휴 비용 0**, 호스팅비 0. PoC 단계에선 생성·평가 모두 **싼 모델 하나**로 돌리고, 품질이 안 나올 때만 생성 노드만 상위 모델로 올린다(모델 계층화).

**속도.** 노트북에서 셀 단위로 만든다. 상태 배선·라우팅·종료조건 같은 **결정적 부분은 LLM 없이 오프라인 스텁**으로 샌드박스에서 먼저 검증하고(가짜 점수를 주입해 루프가 제대로 도는지 확인), 실제 LLM 호출이 필요한 부분만 사용자 맥에서 실행한다. 이러면 배선 버그를 공짜로 잡고, 유료 호출은 로직이 맞다고 확인된 뒤에만 쓴다.

---

## 8. 빠른 검증 계획

검증의 질문은 **"루프가 실제로 점수를 올리는가"** 하나다.

1. 공고 1건 + 문항 1개 + 경험 3~4개로 끝에서 끝까지 1회 실행.
2. 재작성 회차별 평균 점수를 기록해 **점수 곡선**을 본다(우상향하는지).
3. 최종 자소서 1건을 사람 눈으로 합격선 판단(골든 기준 1건 마련).

**성공 기준:** 3회 안에 평균 4.0에 닿거나, 닿지 못해도 점수가 분명히 오르다가 정체 조건에서 **스스로 멈추는** 동작이 확인되면 가설은 산 것으로 본다.

---

## 9. 솔직한 한계 (과장 금지)

- **자기편향:** LLM이 자기 글을 후하게 볼 수 있다. 엄격 루브릭·감점 우선·온도 분리로 줄이지만 없애진 못한다. 1차 검증엔 사람 1건 대조를 병행한다.
- **입력이 상한:** 경험 입력이 부실하면 결과도 부실하다. 좋은 경험을 끌어내는 질문 설계는 별도 과제다.
- **정체 종료는 타협:** 비용을 막는 대신 "조금 더 돌리면 좋아질 글"을 일찍 멈출 수 있다. 임계값(0.3)은 실측 후 조정 대상.

---

## 10. 2차 확장 (골격 재사용)

- **면접 지원:** 최종 자소서 + jd_analysis를 입력으로 예상질문→꼬리질문→모의 Q&A 노드를 같은 에이전트에 추가. 새 인프라 불필요.
- **세션 간 학습("쓸수록 성장"):** 고득점 재작성과 심사위원이 **반복해 지적한 패턴**을 모아 다음 생성의 예시(few-shot)·루브릭 보강에 재사용. 1차 골격이 검증된 뒤 저장 계층만 얹으면 된다.
